## 10.10 회의록

10.10 카카오톡에서 진행할 회의사항

### 개발방법론 설정에 관한 건

필요성: 현재까지 아직 개발이 원활하게 진행되고 있지 않음  
원인: 작업 목표가 불분명하고 역할 책임자가 명확치 않음
해결안 제시: 스크럼 도입

-   스크럼 도입 건의
    -   이유
        -   그때그때 해야할 작업이 다르고, 작업의 중요도가 달라지는 만큼 효율적인 작업의 분배가 필요함
        -   짧은 주기로 개발 상황을 점검하고 조정할 수 있음
        -   팀원간 소통을 원활히 함
        -   역할과 책임을 명확히 함
    -   한계
        -   팀원 각자의 사정이 있어 매일 대면 회의는 어려움
        -   작업에 대한 압박감이 느껴질 수 있음
        -   소규모 인원이므로 역할 분담이 제한적이고, 유연한 역할 수행이 필요함
    -   대안
        -   스크럼의 특징을 살리되, 팀 상황에 맞게 조정하여 적용
        -   매일 대면 회의 대신, 카카오톡 등의 메신저를 활용하여 진행 상황을 공유
        -   책임자를 지정하되, 장애가 발생할 경우 보류할 수 있도록 하며, 팀원 모두가 역할을 유연하게 수행
        -   주간 회의를 통해 전체 진행 상황을 점검하고, 필요시 조정
    -   결론
        -   스크럼을 도입해보고, 팀원들의 피드백을 받아보고자 함
        -   다음 회의에서 스크럼 도입에 대한 의견을 수렴하고, 필요시 조정하고 싶음

### DB 설계에 관한 건

프로젝트가 점점 구체화 됨에 따라 DB 설계가 필요할 것으로 예상됨  
현재 백엔드 팀에서 DB table 구상을 진행중이나, 함께 논의하는 것이 필요하지 않을까 생각됨  
따라서 다음 회의에서 DB 설계에 대한 논의를 진행하고자 건의함

## 9.23 회의록

9.23 Topcit 특강 전 회의사항

### Git 관리 방법 변경

Git 관리를 `front`, `back` 브랜치로 나누기로 결정됨

-   `main > dev > [front | back]` 구조로 설정
-   하위 브랜치는 `front/{브랜치 타입}: {내용}` 등으로 설정함
-   자세한 브랜치 전략은 별도 작성 예정

### 역할 재정의

역할을 일부 재정의함  
백엔드의 작업이 비교적 적을 것으로 예상되어, 백엔드 팀의 논의 후 1명은 프론트로 넘어올 가능성 있음

-   프론트
    -   컴포넌트(기능, 요소) 개발
    -   UI(CSS) 처리
-   백엔드
    -   API개발 / DB

### 프로젝트 진행사항

프로젝트 진행을 위해 `요구사항분석`이 필요하나, 일단 당장 할만한 것들에 대해 논의함  
상세는 `금요일 캡스톤 강의 이후 회의` 중 결정

-   프론트: 웹 화면 디자인, 반드시 필요한 컴포넌트 개발(버튼 등)
-   백엔드: REST API 더미데이터 넣고 구현
